Data-viz usage rulesorder · pairings · accessibility · fallbacks

01Use the categorical palette in order

Always start at position 1 and add hues sequentially. The order is engineered for distinguishability — positions 1 + 2 are maximally distinct, 1 + 3 next, and so on. Skipping or reordering compromises clarity.

02Cap categorical at 8 series

Beyond 8 series, even the best palette starts to blur. If you have more than 8 categories, group the smallest into "Other", switch to a small-multiples layout, or use a different chart type (e.g. a bar chart sorted by value instead of legend-heavy stacked).

03Pair only with neutral chrome

Axes, gridlines, tick labels, and tooltips use the grey ramp — never one of the categorical hues. The vibrant palette pops because the chrome around it stays quiet. Axis text is --sl-grey-700; gridlines are --sl-grey-200; chart background is white or --sl-grey-100 for sunken regions.

04Add a redundant encoding

Color alone is not enough. Pair color with a second encoding — direct labels on lines, patterns/dashes on overlapping series, or icons in legends. This handles colorblind users and B&W printing without rework.

05Don't mix palettes

One palette per chart. A vibrant-categorical chart with one semantic-red series reads as "this one failed". If you need to highlight one series, use weight or annotation, not a different palette.

06Same identity, same color across views

If "Snaplex A" is position 1 (electric indigo) in one chart, it must be electric indigo in every chart on the page. Bind color to identity, not to chart order. This mirrors the avatar-color rule in the table spec.
