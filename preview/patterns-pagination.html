which to use
Table pagination: tables in Manager / Monitor. Matches the Table component exactly — page select + prev/next + rows-per-page. Load more: activity logs, run history. Infinite: feed-shaped UI like the snap palette search; never use for actionable tables. Never mix on one screen.
table
Single canonical control: [‹] Go to page [N ▾] of T [›] with rows-per-page select on the right. Default 25 rows; options 25 / 50 / 100. No numbered page buttons, no first/last. Prev/Next are 28×28 ghost arrows.
load more
Centered secondary button below last row. Label includes batch size ("Load 50 more events"). Show running count beneath ("50 of ~2,400 events"). Estimated totals are fine — exact counts are expensive.
infinite
Auto-trigger when the sentinel row hits the viewport. Show a small spinner row beneath: "Loading 12 more snaps…". Never auto-load without a visible affordance. Always render an end-of-list marker when exhausted.
url state
Table: persist ?page=N&rows=25. Load-more / infinite: persist the cursor, restore to nearest loaded chunk on back nav.
errors
If a fetch fails, replace the loader with an inline retry: "Couldn't load more · Retry". Don't break the rest of the loaded list.
a11y
Table pagination is wrapped in <nav aria-label="Pagination">. Prev/Next arrows use aria-label. Page-select is a real select. Load-more button announces label. Infinite must announce "N new items loaded" via aria-live="polite".
