Foundations

Accessibility

JellyRoll is built to be usable by everyone — whether navigating with a mouse, keyboard, or screen reader. These guidelines apply to all SnapLogic surfaces.

WCAG 2.1 AA Target Conformance

The four POUR principles

Perceivable
Information and UI components must be presentable in ways users can perceive — including non-visual alternatives.
Operable
UI components and navigation must be operable by keyboard, not just a pointer device.
Understandable
Information and operation of the UI must be understandable — clear language, predictable behavior.
Robust
Content must be interpreted reliably by assistive technologies, including current and future user agents.

Color and contrast

Never use color as the only means of conveying information or requesting an action. Always pair color with a label, icon, or pattern.
Pairing Use Ratio WCAG AA
Indigo-1000 on White Body text, headings 17.5:1 AA Normal & large text
Blue-600 on White Interactive / link text 7.2:1 AA Normal & large text
White on Blue-600 Primary button label 7.2:1 AA Normal & large text
Grey-500 on White Subtle / placeholder text 4.6:1 AA Normal text minimum
Blue-600 on Grey-300 Avoid — insufficient for small text 2.9:1 Fails normal text

Keyboard navigation

Every interactive element in JellyRoll must be reachable and operable by keyboard alone. All focus indicators use a 3px teal halo at 35% alpha — never remove the outline without providing an equivalent replacement.

Key Action Applies to
Tab Move focus to next interactive element All focusable elements
Shift + Tab Move focus to previous interactive element All focusable elements
Enter Activate button or link; submit form Buttons, links, form submit
Space Toggle checkbox, radio, or toggle; activate button Checkboxes, toggles, buttons
↑ ↓ Navigate items in a list, menu, or select Dropdowns, menus, radio groups
← → Navigate tabs, segmented controls, sliders Tabs, segmented controls
Esc Close modal, dismiss popover, cancel action Modals, popovers, drawers, tooltips
Home / End Jump to first or last item Lists, tables, sliders

Semantic HTML and ARIA

Use native HTML elements before reaching for ARIA. A native <button> gives you keyboard support, focus management, and screen reader semantics for free.

Do

Use semantic elements that describe their purpose.

<button type="button">Save</button> <nav aria-label="Primary">...</nav> <main>...</main>

Add aria-label when visible text is absent (icon-only buttons).

<button aria-label="Close dialog"> <svg ...>...</svg> </button>
Don't

Don't use non-semantic elements as interactive controls.

<div onclick="save()">Save</div> <span role="button">Close</span>

Don't use ARIA roles to paper over missing semantics when a native element exists.

<div role="button" tabindex="0" onclick="...">Save</div>

Forms and inputs

Every input must have a visible label
Placeholder text disappears on focus and is not a substitute for a label. The only exception is search fields, where the magnifying glass icon serves as a visible label — but even these require a visually hidden <label> for screen readers.
Associate labels with inputs using for and id
Clicking a properly associated label should move focus to the control. Never use aria-label to replace a visible label on a standard form field.
Surface validation errors inline and programmatically
Use aria-describedby to link the error message to its field. Set aria-invalid="true" on the input when it fails validation. Move focus to the first error on form submit.
Group related inputs with <fieldset> and <legend>
Required for radio and checkbox groups. The legend text is read by screen readers before each option in the group, providing essential context.

Images and icons

Meaningful images require descriptive alt text
Describe the content and purpose of the image, not its appearance. Keep alt text concise — one or two sentences. Do not start with "Image of" or "Photo of".
Decorative images use alt="" and aria-hidden="true"
Decorative images, dividers, and background textures should be invisible to screen readers. An empty alt="" on an <img> tells screen readers to skip it entirely.
Standalone icon buttons need an accessible label
For icon-only buttons, use aria-label on the <button> element. For inline informational icons (status indicators, help icons), use a <title> inside the SVG or aria-label on the surrounding element.

Focus management

Trap focus inside open modals and drawers
When a modal opens, focus should move to the first focusable element inside it. Tab cycling must remain within the modal until it is closed. On close, return focus to the trigger element that opened it.
Never suppress the visible focus ring
The JellyRoll focus ring is a 3px teal halo. Use :focus-visible to show it for keyboard users while hiding it on mouse click. Setting outline: none without a replacement fails WCAG 2.4.7.
Maintain a logical focus order
Focus order must follow the visual reading order (left-to-right, top-to-bottom). Avoid tabindex values above 0, which disrupt the natural DOM order and confuse keyboard users.

Testing checklist

Keyboard-only navigation
Unplug the mouse. Can you reach every interactive element with Tab/Shift-Tab? Does Esc close modals? Do arrow keys work in menus and selects?
Screen reader smoke test
Test with VoiceOver (macOS/iOS) or NVDA (Windows). Navigate by headings, landmarks, and form controls. Confirm all interactive elements announce their role, name, and state.
Automated scan with an axe-based tool
Run axe DevTools, Deque's browser extension, or Lighthouse accessibility audit on every new page or component. Fix all critical and serious violations before shipping.
200% zoom test
Increase browser zoom to 200%. Content must not overflow or overlap. All functionality must remain available without horizontal scrolling (WCAG 1.4.4).
